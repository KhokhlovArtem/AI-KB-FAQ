## Задача

| Параметр | Решение |
|----------|--------|
| **База знаний** | Текстовые документы с четкой структурой, доступ по API GitLab |
| **Объем** | Пока неизвестен, архитектура должна масштабироваться |
| **Время ответа** | Оперативно, 30–60 секунд |
| **LLM** | Внешние API: YandexGPT или GigaChat |
| **Точка входа** | Telegram-бот (в перспективе — любые интерфейсы) |
| **Отказ LLM** | Если нет релевантных фрагментов в базе знаний → формируется письмо в поддержку |
| **Пользователи** | Есть идентификатор (TG user_id) для связки |

## Техническая архитектура (концептуально)

### 1. Компоненты

```
┌─────────────────┐     ┌─────────────────────────────────────────────────┐
│  Telegram Bot   │────▶│              Orchestrator (API)                 │
└─────────────────┘     │  - Принимает сообщения                         │
                        │  - Управляет сессиями                          │
                        │  - Решает: RAG или тикет                       │
                        └─────────────────────────────────────────────────┘
                                        │
            ┌───────────────────────────┼───────────────────────────┐
            ▼                           ▼                           ▼
┌─────────────────────┐   ┌─────────────────────┐   ┌─────────────────────┐
│   Pipeline: RAG     │   │   Pipeline: Тикет   │   │   GitLab API       │
│                     │   │                     │   │   (выгрузка БЗ)    │
│ 1. Embedding        │   │ 1. Собрать контекст │   └─────────────────────┘
│ 2. Поиск в векторной│   │ 2. Сформировать     │
│    БД               │   │    email            │   ┌─────────────────────┐
│ 3. Запрос к LLM     │   │ 3. Отправить на     │   │   Векторная БД     │
│    (Yandex/Giga)    │   │    support@...      │   │   (Qdrant / PG)    │
│ 4. Ответ пользова-  │   └─────────────────────┘   └─────────────────────┘
│    телю             │
└─────────────────────┘
```

### 2. Процесс обработки запроса

**Шаг 1 — Получение вопроса**  
Telegram-бот отправляет `user_id` и текст в Orchestrator.

**Шаг 2 — Поиск релевантных фрагментов**  
- Текст пользователя эмбеддится (модель от Yandex/Giga или отдельная embedding-модель)  
- Выполняется поиск в векторной БД по косинусной близости  
- Задается **порог релевантности** (например, `0.75`)

**Шаг 3 — Ветвление**

| Результат поиска | Действие |
|------------------|---------|
| Есть фрагменты выше порога | → RAG-пайплайн: формируем промпт (контекст + вопрос), вызываем LLM, возвращаем ответ пользователю |
| Нет фрагментов выше порога | → Тикет-пайплайн: формируем письмо в поддержку, пользователю отправляем сообщение "Не нашли ответ в базе знаний, передали вопрос специалистам" |

**Шаг 4 — Формирование письма в поддержку**  
Письмо содержит:
- `user_id` (или ссылку на диалог в TG, если поддержка тоже в TG)
- Исходный вопрос пользователя
- (опционально) то, что было найдено ниже порога — для контекста
- Временная метка

Отправка через SMTP или API почтового сервиса.

### 3. Обновление базы знаний

- Периодический (или по webhook из GitLab) запуск пайплайна индексации
- Забираем документы через GitLab API
- Разбиваем на чанки (размер: ~500–1000 токенов, с overlap)
- Генерируем эмбеддинги
- Сохраняем в векторную БД вместе с метаданными (путь к файлу, ссылка на GitLab, дата обновления)

### 4. Выбор технологий

| Компонент | Рекомендация |
|-----------|--------------|
| **Orchestrator** | Python (FastAPI) — гибкость, богатая экосистема LLM |
| **Telegram-бот** | `python-telegram-bot` или `aiogram` |
| **Векторная БД** | Qdrant (облачная или self-hosted) или pgvector если уже есть PostgreSQL |
| **Embedding-модель** | `text-embedding-ada` от YandexGPT или `embeddings` от GigaChat (для единообразия экосистемы) |
| **LLM API** | YandexGPT или GigaChat (выбираем одну, либо делаем адаптер для обеих) |
| **Индексация** | Celery + Redis для фоновых задач (если объем большой) |
| **Отправка писем** | SMTP (корпоративная почта) или SendGrid / Unisender API |

### 5. Ключевые вопросы для дальнейшего проектирования

1. **Порог релевантности** — нужно будет подбирать экспериментально на тестовой выборке вопросов.
2. **Авторизация** — если база знаний содержит чувствительные данные, нужно убедиться, что GitLab API работает с токенами и доступ к векторной БД тоже ограничен.
3. **Мультиинтерфейсность** — архитектура с Orchestrator позволяет легко добавить веб-чат, виджет или другие каналы без изменения логики.
4. **Логирование** — все отказы (переход в тикет) должны логироваться для анализа, какие вопросы не покрываются базой знаний.
